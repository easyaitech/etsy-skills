# Schema：operator 中央运营表 + Orders 物流字段

两处落点，泾渭分明：

- **operator 中央运营表**（跨租户、operator 自有，**不含任何客户明细**）：全局月配额账本 + 巡检运行锁/状态。这是 D2 配额护栏和 D5 巡检锁的合法落点（决定 #4"不新建每租户表"指的是别为跟踪队列建每租户表；一张 operator 运营表是另一回事）。
- **店铺总 Base 的 `Orders 订单` 表**（每租户，orders-customers 拥有）：补一组物流字段。本 skill 只写这些物流字段，不碰 `状态`/`SOP 阶段`。

---

## 一、operator 中央运营表

> 用 `lark-base` 在一个 **operator 管理 Base**（不是任何租户的店铺总 Base）里建下面两张表。机器人/Hermes 对该 Base 的权限独立于租户 Base。**只存计数、锁、运行元数据；不存订单号、跟踪号、客户信息。**

### 表 1：`配额账本`

按 (月份, 租户) 记注册消耗，用于**租户级归因**（为 v2 每租户上限留数据）。**全局配额不靠本表**——实测 17TRACK 有 `/getquota` 实时返回 `quota_remain`，全局阈值/耗尽判定直接查它（见 [17track-adapter §二](17track-adapter.md) 与 [sweep-contract §⑤](sweep-contract.md)）；`租户=__GLOBAL__` 行可选，仅作本地留痕。v1 单 workspace 时本表可极简甚至缓建，巡检锁（表 2）才是 v1 必需。

| 字段名 | 类型 | 说明 |
|---|---|---|
| `账本键` | 单行文本（**主键**） | `{YYYY-MM}:{租户标识}`，如 `2026-06:teacup-shop`；全局行用 `2026-06:__GLOBAL__` |
| `月份` | 单行文本 | `YYYY-MM`（自然月，17TRACK 免费额度按自然月重置的口径——上线前实测确认重置口径） |
| `租户` | 单行文本 | 租户标识（workspace slug / 店铺名）；`__GLOBAL__` 表示全局汇总行。**不是客户 PII** |
| `本月注册数` | 数字 | 该 (月份,租户) 已成功 register 的单号数；register 成功后 +1 |
| `配额上限` | 数字 | 仅作参考/覆盖；**实时上限以 `getquota.quota_total` 为准**（本账户实测 200，不是免费档 100）。租户行 v1 留空，v2 填每租户上限 |
| `告警阈值` | 数字 | `quota_remain` **低于**此值即推飞书告警，默认 20（对照 getquota 实时余量，非自维护计数）；可调 |
| `最后告警时间` | 日期时间 | 上次推过配额告警的时间，避免重复轰炸 |
| `状态` | 单选 | 正常 / 近阈值 / 已耗尽 |
| `更新时间` | 日期时间 | 最后一次写本行的时间 |

> v1 只强校验**全局行**（决定 #2 集中 key、决定 D2）。租户行同时维护（计数器按租户可分），为 v2 的每租户上限留好数据，不需额外迁移。

### 表 2：`巡检运行`

巡检单实例锁 + 运行态 + 中断恢复依据（D5）。

| 字段名 | 类型 | 说明 |
|---|---|---|
| `运行键` | 单行文本（**主键**） | 固定 `sweep-lock`（单例）；如需按租户分锁则 `sweep-lock:{租户}` |
| `执行锁` | 单行文本 | 占用时写本轮唯一令牌 `logistics-tracking:{run_id}`；完成/失败/放弃后清空 |
| `锁状态` | 单选 | 空闲 / 运行中 |
| `锁占用时间` | 日期时间 | 写锁的时间；超过 `锁超时` 视为僵尸锁可抢占（默认 30 分钟） |
| `上次开始` | 日期时间 | 上轮巡检开始 |
| `上次结束` | 日期时间 | 上轮巡检结束 |
| `上次结果` | 多行文本 | 处理数 / 注册数 / 推送数 / 失败数 摘要 |
| `本轮处理数` | 数字 | 用于中断后核对 |

**占用规则**（套 [social-publisher publishing-queue-contract](../../social-publisher/references/publishing-queue-contract.md) §占用规则）：生成唯一令牌 → 条件更新（仅当 `锁状态=空闲` 或锁已超时）写 `锁状态=运行中`+令牌+时间 → 重读确认令牌是本轮的，不成立就停、不巡检 → 工具无法条件更新且无法证明唯一占用时，巡检只能**单实例**手动跑，不并发 → 完成后清锁。中断遗留的"运行中"超时后下轮可抢占；抢占后对"注册中"的单号**直接重 register**（实测幂等不重扣，返回 -18019901 即已注册；见 sweep-contract §②）。

## 二、`Orders 订单` 表补物流字段

> 在店铺总 Base 的 `Orders 订单`（orders-customers 拥有）里**补**下列字段。建表时缺哪个补哪个，给用户确认后再补；不改动 orders-customers 已有字段的语义。carrier 提示**复用已有的 `快递公司` 字段**，不新建。

### 物流状态字段组

| 字段名 | 类型 | 说明 |
|---|---|---|
| `物流状态` | 单选 | **中性枚举**（见 [17track-adapter](17track-adapter.md) §状态映射）：未发出 / 已揽件 / 在途 / 到达待取 / 派送中 / 已签收 / 异常 / 退回 / 挂起待人工。本 skill 唯一写者 |
| `最新轨迹` | 多行文本 | 最新一条轨迹的描述（地点+事件），只存最新摘要不存全程长文 |
| `物流更新时间` | 日期时间 | 17TRACK 返回的该轨迹时间（不是本地查询时间） |
| `物流签收日期` | 日期 | `物流状态=已签收` 时写入；这是 orders-customers 接管的真实签收信号 |

### 跟踪运行字段组（支撑幂等/错误/去重，D5/D7/#10/#12）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `17track注册状态` | 单选 | 未注册 / 注册中 / 已注册 / 注册失败。幂等注册的依据（D5）。**空值视同未注册**（覆盖接入前存量单） |
| `物流注册时间` | 日期时间 | 成功 register 的时间；TTL 兜底——黑洞件（NotFound、无承运商事件、`物流更新时间` 为空）靠它判超期挂起（见 sweep-contract §⑦） |
| `物流查询尝试次数` | 数字 | 暂时错重试计数；≥2 转人工（D7） |
| `下次重试时间` | 日期时间 | 暂时错退避后的下次可查时间 |
| `上次查询时间` | 日期时间 | 本地上次成功查询的时间（运维用） |
| `查询失败原因` | 单行文本 | 永久错/失败时记原因（无效号/不支持/额度耗尽…），不编造 |
| `物流通知事件标记` | 单行文本 | last-notified 事件 hash / id，推送去重用（#12）——状态没新跃迁不重复推 |
| `快递公司` | 单行文本 | **只读**——orders-customers 拥有/填写（发货字段，base-schema.md:64）；本 skill 仅**读**作 17TRACK carrier hint，**不写**。为空靠自动识别，识别失败在 orders 侧人工补 |

### 视图建议

- **待跟踪** — `状态=已发货 ∧ 跟踪号非空 ∧ 物流状态∉{已签收,退回,挂起待人工}`（这就是巡检的待查队列）
- **物流异常** — `物流状态∈{异常,挂起待人工}` 或 `查询失败原因` 非空 — 人工要看
- **今日新签收** — `物流签收日期 = 今天`（给 orders-customers 衔接用）

## 三、与 orders-customers 的衔接（D3 单写者 + #11 最终一致）

- 本 skill 写 `物流状态=已签收` + `物流签收日期`，**到此为止**；不翻 `状态`、不动 `SOP 阶段`、不发买家消息。
- orders-customers 的"待签收跟进"逻辑（见其 [base-schema](../../orders-customers/references/base-schema.md) §视图 与 order-fulfillment-sop §6）改为**读 `物流状态=已签收` 作为真实签收信号**，替掉原来按 `签收跟进日期` 估算的口径，由它拥有 `状态→已签收`、`SOP 阶段`、评价引导（人工确认）。
- 这是**最终一致**：本 skill 推飞书通知 + 写字段，orders-customers 下一轮处理时扫到接管，不是同步触发。可接受的延迟，换来干净的单写者边界。
