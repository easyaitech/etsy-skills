# 巡检契约（模式 C）

模式 C 把"已发货未签收"的订单批量跟进一轮。v1 手动触发跑单 workspace；v2 才接 Hermes 无人值守 cron + 多 workspace 枚举（见 TODOS，先 spike 调度机制）。

## 巡检流水线

```text
① 占锁（中央表 巡检运行）──失败(已有实例/未超时)→ 停，不并发
        │ 成功
        ▼
② 中断恢复：上轮遗留"注册中"的单号直接重 register（幂等不重扣，-18019901=已注册）
        │
        ▼
③ 筛待查队列（对 Orders 一次筛选读）：
   状态=已发货 ∧ 跟踪号非空 ∧ 物流状态∉{已签收,退回,挂起待人工} ∧ (下次重试时间为空 或 ≤现在)
        │
        ▼
④ 分批（每 40 个跟踪号一批，≤3 请求/秒）逐批处理：
        ├─ 新号(17track注册状态=未注册，空值视同未注册→含接入前存量单) → 配额护栏 → register → 标 已注册
        ├─ 已注册 → gettrackinfo standard（免费）
        └─ 适配器返回 → 按中性枚举映射
        │
        ▼
⑤ 批量回写 Orders 物流字段（字段级 patch，不整条覆盖）
        │
        ▼
⑥ 去重推送：物流状态有跃迁 → 推飞书给运营者（按 物流通知事件标记 去重）
        │
        ▼
⑦ 终态处理：已签收→写签收日+推送+交回 orders；退回/挂起→停跟+标人工
        │
        ▼
⑧ 清锁 + 写 巡检运行（上次开始/结束/结果摘要）
```

## ① 单实例锁

按 [social-publisher §占用规则](../../social-publisher/references/publishing-queue-contract.md) 套用，落在中央表 `巡检运行` 行（见 [base-schema](base-schema.md) §表2）：生成 `logistics-tracking:{run_id}` 令牌 → 条件占用（仅当空闲或锁超时）→ 重读确认 → 占不到就停。**lark-base 若无法保证条件更新的原子性，巡检只能单实例手动跑，绝不并发启动多个 runtime。**

## ② 中断恢复（D5，spike 后简化）

实测 register **幂等且不重扣**（已注册返回 `-18019901`）。所以抢占僵尸锁（超 `锁超时`，默认 30 分钟）后，对遗留 `17track注册状态=注册中` 的单号**直接重 register 即可**：返回 `-18019901` → 标 `已注册`（本就注册过）；返回 `accepted` → 标 `已注册`（这次才真上）。不必单独探测、不会重扣额度。`17track注册状态` 字段保留只为可读 + 省一次被限速的调用，不再是配额安全命门。

## ③ 待查队列

一次 `lark-base` 筛选读取（不逐单读，避免 N+1，D9）。下次重试时间未到的暂时错订单本轮跳过。

## ④ 幂等注册 + 分批（D5/D9）

每单 `17track注册状态` 状态机：

```text
未注册 ──register 成功──▶ 已注册 ──gettrackinfo──▶ (轮询，免费)
   │                         ▲
   ├─register 暂时错→ 留未注册，下次重试      │
   ├─register 永久错→ 注册失败 + 查询失败原因 + 标人工
   └─register 额度耗尽→ 留未注册 + 触发⑥配额降级（不重试）
   注册中（崩溃遗留）──直接重 register──▶ 已注册（-18019901 或 accepted 均归位，不重扣）
```

分批：每批 ≤40 个跟踪号、整体 ≤3 请求/秒。register 与 gettrackinfo 都受此限（维度待 §spike 实测）。

## ⑤ 配额护栏（D2）★（spike 后改用 getquota）

全局配额以 **`/getquota` 实时返回的 `quota_remain` 为准**（实测可用、免费），不自维护全局计数器。**护栏要在注册循环内持续生效，不能只在开头查一次**——否则一次手动 sweep 能把共享额度一路注册到耗尽：

- **单轮上限**：巡检开始查一次 `quota_remain`，估算本轮待注册新号数；设 `单轮注册上限`（中央表配置，默认 50）。本轮新号超过 `min(单轮注册上限, quota_remain - 告警阈值)` 时，只注册前 N 个、其余留下轮，并推飞书"本轮新号过多，已分批"。
- **批内重查**：**每注册一批（≤40）前重查 `quota_remain`**（standard 查询不耗额度，重查便宜）；低于阈值即停注册新号，绝不凭开头那次读数注册到底。
- `quota_remain ≤ 告警阈值`（默认剩 20，**是绝对余量不是百分比**，对照 `getquota.quota_total`）且本月未告警 → 推飞书"17TRACK 配额剩 N（共 M），接近上限"，写中央表 `最后告警时间`。仍可注册但醒目。
- `quota_remain ≤ 0`（耗尽）→ **停注册新号**，推飞书告警；已注册号照常 gettrackinfo（免费），新号留 `未注册` 等下月重置或人工提级。**绝不静默失败。** 兜底：register 被配额码直接拒也按耗尽处理，不当暂时错重试。
- register 成功后给该租户行 `本月注册数 +1`（租户级归因，为 v2 每租户上限留数据）。

> 共享 key = 共享故障域：一个旺租户可能吃光全局额度连坐其他租户。v1 靠全局护栏（getquota 阈值，不静默死）兜底；v2 加每租户月上限（中央表租户级计数已就绪）。

## ⑥ 去重推送（D4/#12）

- 只在 `物流状态` 发生**跃迁**（与上次记录不同），或出现新的关键轨迹事件时，才推飞书给**运营者本人**。
- 用 `物流通知事件标记`（last-notified 事件 hash/id）去重：本次事件标记与已存相同则不推。状态没变不推、同一异常不每天炸。
- 推送内容：订单号 + 中性 `物流状态` + 最新轨迹 + 时间。**只发给运营者，不发买家。**

## ⑦ 终态 + 交接（D3/#11/#7）

- **已签收**（终态）：写 `物流签收日期`，推飞书"订单 X 已签收"，从待查队列移出。**不翻 `状态`/`SOP 阶段`、不发买家消息**——orders-customers 的"待签收跟进"读 `物流状态=已签收` 接管（最终一致）。
- **退回**（终态）：停跟，标人工。
- **挂起待人工**（终态，TTL）：满足任一即置 `挂起待人工`、停每日轮询、标人工——① `异常` 持续超阈值；② `物流更新时间` 距今 > `无更新阈值`（默认 60 天）；③ **`物流注册时间` 距今 > 无更新阈值 但仍无任何承运商事件**（NotFound/未发出、`物流更新时间` 为空的黑洞件，靠注册时间兜底，否则永不挂起）；④ 承运商识别失败。避免异常件/黑洞件永久轮询。
- `异常` 但未超 TTL：保留在队列继续跟，并按去重规则推一次告警。

## ⑧ 错误分类（D7）

见 [17track-adapter §四](17track-adapter.md)。暂时错退避重试（尝试次数<2，写下次重试时间）；永久错记原因+标人工不重试；额度耗尽走 ⑤ 降级不重试。

## dry-run / test（上线前必跑）

见 [runtime-setup §dry-run](runtime-setup.md)。test 模式跑完整 ①–⑧ 流水线，但：用真实测试单号、**不推飞书、不翻真字段、写 scratch 区**，专门验证两个 ★CRITICAL 分支（⑤配额耗尽降级、②中断恢复）和去重/锁/状态映射。dry-run 通过再切真跑。
