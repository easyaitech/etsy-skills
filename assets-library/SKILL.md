# 资产库 (Assets Library)

这个 skill 维护 Etsy 店铺的视觉与素材资产，核心是**双层架构**：
- **物理层（文件夹）**——飞书云空间，每个文件唯一存放，按"原料 / 成品 / 受限"粗分
- **语义层（素材索引 Base）**——一行一个素材，多选标签 + 关联其他 Base，承载所有"网状归类"

文件夹只回答"这个文件物理在哪"，Base 回答"它属于哪些 SKU / 渠道 / 客户 / 用途"。两者通过"文件链接"字段连接。

**对外的实操接口**：飞书云空间（`lark-drive`） + 素材索引 Base（`lark-base`） + 项目根目录的 BRAND.md（视觉原则）。

---

## 依赖关系

| 来源 | 提供什么 | 怎么用 |
|---|---|---|
| `./BRAND.md` § 视觉原则 | 整体气质 / 色彩 / 排版构图 / 视觉禁区 | 归档新摄影 / 处理新模板时用作"是否合规"自检；自检结果写进素材索引 Base 的"BRAND 合规"字段 |
| 商品 Base | SKU 列表 | 素材索引 Base 的"关联 SKU"字段直接关联；商品 Base 通过反向查询能看到该 SKU 的全部素材 |
| 订单 Base | 订单号 | 客户拍摄 / 客户定制类素材通过"关联订单"字段挂上 |
| 客户 Base | 客户列表 | UGC 类素材通过"关联客户"字段溯源授权 |

## 三种执行模式

### 模式 A：建库（首次建立双层骨架）

**进入条件**：飞书云空间还没有规范的资产目录，或还没有素材索引 Base。

**执行步骤**：
1. 先按 [asset-index-base-schema.md § 何时不需要建这张 Base](references/asset-index-base-schema.md#何时不需要建这张-base) 与用户确认是否真的需要 Base——若否，本模式只做物理层
2. 读 `references/folder-structure.md` 与 `references/asset-index-base-schema.md`，对齐双层结构
3. **物理层**：用 `lark-drive` 在飞书云空间创建根文件夹 `{店铺名}-素材库`，按推荐骨架建子目录（视频暂不做可跳过）
4. **语义层**（如需要）：用 `lark-base` 在与商品/订单/客户 Base 同一个空间创建 `{店铺名}-素材索引` Base，按 schema 建字段（关联字段必须指向已有的三张 Base）和推荐视图
5. 落盘后告诉用户：根目录链接 + Base 链接（如建了） + 简要使用约定

> **不要替用户上传任何文件**——建库阶段只创建空骨架。让用户自己把已有素材分批迁过来；迁的过程中可以求助本 skill 给命名建议 + 录索引。

### 模式 B：归档新素材（双写：文件夹 + 素材索引 Base）

**进入条件**：用户上传了新摄影 / 视频 / 设计 / 模板 / 客户素材，需要按规范归档。

**执行步骤**：
1. 读 `references/naming-convention.md`：确认命名规则
2. 读 `references/asset-types.md`：识别素材类型 → 应归到哪个粗分目录
3. 给出**目标路径 + 标准化文件名**建议：
   ```
   素材库/1. 摄影/by-SKU/TEACUP-001/raw/2026-05-05_TEACUP-001_morning-light_001.jpg
   ```
4. 用户确认后，用 `lark-drive` 创建子目录（如缺）+ 上传 / 移动文件，拿到飞书云空间文件链接
5. **录入素材索引 Base**——按 [asset-index-base-schema.md § 录入约定](references/asset-index-base-schema.md#录入约定模式-b-归档时执行) 执行；用 `lark-base` 新增一行，**录入前列出将写入的字段值清单等用户确认**
6. 商品 Base 该 SKU 行通过"关联 SKU"反向就能看到本条素材；商品 Base 的"照片链接"跳转字段如需手动维护，参 [asset-types.md § 与 listing-catalog 的协作](references/asset-types.md#与-listing-catalog-的协作)

> **视觉合规检查**：归档前如果素材明显违反 BRAND.md 视觉禁区（刻板符号、高饱和、AI 拼贴感等），主动提示——不替用户拒绝归档，结论写进 Base 的"BRAND 合规"字段（不通过时理由写"备注"）。

> **客户拍摄 / UGC 类**："公开授权"默认 `待沟通`，必须挂"关联订单 + 关联客户"；授权拿到前不要勾任何渠道用途标签。

### 模式 C：查找（先 Base 后文件）

**进入条件**：用户要找素材（"某 SKU 的图"、"那个新春模板"、"Logo PNG 透明底"、"客户 UGC 里能发 Pinterest 的"）。

**优先用素材索引 Base 查**——它是为多维检索而生：

| 用户问 | Base 视图 / 筛选条件 |
|---|---|
| "TEACUP-001 所有摄影" | 视图"按 SKU 分组" → 找 TEACUP-001 |
| "能发 Pinterest 的图" | 视图"Pinterest 候选"，或筛 用途标签 ⊇ Pinterest AND 公开授权 = 已授权 |
| "客户拍的、能用作评价的" | 视图"客户 UGC（已授权）" → 筛 素材类型 ⊇ 客户拍摄 |
| "ORDER-2026-001 客户给的所有定制参考" | 筛 关联订单 = ORDER-2026-001 |
| "竖版视频母版" | 筛 素材类型 ⊇ 视频母版 AND 比例 ⊇ 9x16 |

**执行步骤**：
1. 用 `lark-base` 按上述模式查 Base，拿到候选素材列表（含文件链接）
2. 必要时用 `lark-drive` 跟进打开文件 / 提供更多元信息
3. 候选多时，按"上传日期"或"BRAND 合规"排序提供给用户

**仅当 Base 不存在或没录入这条素材时**，回退到 `lark-drive` 直接搜文件名 + 引用 `references/folder-structure.md` 推断目录位置。这种情况下顺手提示用户"这条素材没在 Base 里，要补录吗？"

## 写入前的硬性约束

1. **文件操作**：上传 / 移动 / 重命名 / 删除前，列出动作清单，**等用户确认**才执行
2. **Base 录入**：新增 / 修改素材索引 Base 行前，列出字段值清单，**等用户确认**才执行
3. **不删除原始素材**：摄影原图（RAW / 高清 JPG）一旦归档，本 skill 不主动删除——容量管理由用户人工决策
4. **不替用户处理图片内容**：本 skill 只管目录、命名、归档、索引；图片裁切、调色、加水印、生成场景图等由用户或其他工具完成
5. **single source of truth**：每个文件在云空间只放**一份**——多归属靠 Base 多选字段表达，不要靠拷贝。详见 [asset-index-base-schema.md § 设计原则](references/asset-index-base-schema.md#设计原则)

## 与其他 skill 的协作

- **shop-foundation**：每次归档后如果发现 BRAND.md 视觉原则需要补充（比如新增"自然光必拍"这种偏好），按 shop-foundation 沉淀流程提议进 BRAND.md
- **listing-catalog**：商品 Base 通过"关联 SKU"反向看到全部素材；详见 [asset-types.md § 与 listing-catalog 的协作](references/asset-types.md#与-listing-catalog-的协作)
- **orders-customers**：客户定制参考图与 UGC 通过"关联订单 + 关联客户"挂上；UGC 授权流程（找客户沟通）由 orders-customers 完成，结果回写到 Base 的"公开授权"字段

## 工作语言

- 与用户对话：中文
- 文件名：英文（避免飞书云空间路径里有中文带来的兼容问题，且便于跨平台使用）
- 目录名：可中英混用（飞书原生支持中文目录名）
