# 沉淀方法

每次沉淀走三步：**识别 → 抽象 → 归位**。

---

## 第一步：识别 — 这条纠正值得沉淀吗？

不是所有纠正都该进 BRAND.md。判别标准：

| 该沉淀 | 不该沉淀 |
|---|---|
| 反映用户对"品牌该是什么"的偏好 | 单点事实错误（错别字、价格写错、产品名拼错） |
| 同类任务下次还会遇到 | 一次性、与品牌无关的修改 |
| 能抽象成原则或边界 | 只是当下心情或具体场景特例 |
| 用户语气里有"我们品牌应该……"、"这不像我们" | 用户只说"改下这个"，没有偏好倾向 |

**判别流程**：
- 用户语气已带明确品牌信号（"不像我们"、"我们品牌应该……"、"这种风格不合适"）→ 直接进入第二步抽象
- 不确定时 → 先问："这次的修改是只改这一处，还是反映了一条普适的偏好？" 让用户自己分类

无论哪种，抽象后的草稿仍要展示给用户确认才能落盘。

---

## 第二步：抽象 — 从具体到原则

抽象的核心：往**深一层**追问"为什么"。

模板：
1. 用户的具体反馈是 X
2. 追问："是因为什么觉得 X 不对？" 或 "你期待的是什么样的？"
3. 得到原因 Y
4. 把 Y 表达成**正向 + 负向**并存的一条原则：「（应该）……；（避免）……」

正向 + 负向并存很重要——只说"该是什么"会留歧义，加上"不该是什么"才能锁住边界。

例子：

| 具体反馈 | 追问后 | 抽象原则 | 归入维度 |
|---|---|---|---|
| "这段文案太冷了" | "希望像和老朋友聊天" | 文案语调以亲近、放松的口吻为主；避免商业感过强的措辞和套话 | 文案语调 |
| "这张图不对" | "太工业感了，缺少手作温度" | 视觉强调手作痕迹与自然质感；避免过于精致、工业化的呈现 | 视觉原则 |
| "客服回复别这么快道歉" | "希望先共情、了解情况再给方案" | 处理负面反馈时优先共情、再了解、最后给方案；不做条件反射式道歉 | 文案语调 |
| "这个绿不太对，太鲜了" | "希望是更沉、更哑光的那种" | 主绿调整为低饱和度、偏哑光（具体色号需用户提供或共同确认） | 视觉原则 |

> 视觉维度抽象时可以保留具体色号；其他维度保持原则级表达。

### 抽象的失败征兆

写完原则后回头看，如果出现以下情况，重写一次：

- **太具体**：包含"这次"、"那个"、"昨天的那张"这类指示词，说明还没跳出当前任务
- **太抽象**：用了"高质量"、"用户友好"这类正确但无信息的词，说明没真正抽出特征
- **没有反面**：只说该是什么，没说不该是什么——歧义未消除
- **和已有原则雷同**：只是换种说法重复——应该合并而非新增

---

## 第三步：归位与写入

1. 提议归入哪个维度的哪个段落（新增原则进"原则"列表；新增禁区进"边界"或"视觉禁区"；与现有原则相关 → 是补充、合并、还是冲突，见下文"冲突处理"）

2. 用 **diff 风格**展示草稿（不要只口头描述）：
   ```
   ### 原则
   - 像友人分享
   - 可文气但不古板
   + 处理负面反馈时优先共情、再了解、最后给方案；不做条件反射式道歉
   ```

3. 落盘按 SKILL.md §写入前的硬性约束执行（含确认、最后更新日期同步、修订日志追加：`YYYY-MM-DD: <维度>新增"<原则简称>"原则（来源：<触发任务>）`）

---

## 冲突处理

如果新原则与已有原则**矛盾**（不是补充，而是冲突），不要默默替换。明确告诉用户：

> 「我注意到你这次的偏好和 BRAND.md 里现有的『XXX』原则有冲突。你希望：
> A. 替换现有原则
> B. 给现有原则加例外条件
> C. 留着我再想想，先不写入」

冲突往往意味着品牌还在演化，要给用户思考空间。**永远不要替 Claude 拍板替换**——底座的权威性靠用户保障，不靠模型自动化。

---

## 节制原则

底座的价值在于**精炼**——下游 Agent 是逐条对照的：

- **单条以一句话为目标**，能容纳"应该……避免……"对偶就够。超过两句、塞进 5+ 个反例词、同时讲多个偏好 → 拆成两条，或把具体禁用词移到"避免说"列表
- **维度内 ≤ 5 条原则**。已达 5+ 时新增前先问用户：「这条要新增、合并进现有某条、还是替换某条？」
