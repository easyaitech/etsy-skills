# SHOP.md 更新方法

> SHOP.md 是事实档案，更新方法和 BRAND.md 完全不同：**没有"抽象"步骤**，事实变了就直接改。

## 三步流程：识别 → 定位 → 更新

### 第一步：识别 — 这是事实变更吗？

不是所有针对店铺信息的话都触发更新。判别：

| 该更新 | 不该更新 |
|---|---|
| 用户明说事实变了（"现在处理时间改成 5 天了"） | 用户只是在某次任务里临时打破规则（"这单急，承诺 2 天发"） |
| 用户读 SHOP.md 时纠正现存字段（"这里写错了，主理人是 X 不是 Y"） | 用户问"你能不能改成……" 的探讨语气，未确认 |
| 用户主动添加新条款（"以后欧盟订单要加 VAT"） | 单点录入错误（错别字、链接错配），可直接改无需审议 |
| 用户更新 About / Announcement 的线上文案 | Agent 一次性写的对外文案（这条进新版本备份，但不替换 About 原文） |

不确定时直接问："这是这次任务的临时调整，还是 SHOP.md 要从此改成这样？"

### 第二步：定位 — 哪个字段？

SHOP.md 有六大块（基础 / 政策 / About / Announcement / 节假日 / 合规），每块下又有具体字段。准确定位到要改的字段，不要扩散修改。

特别注意：
- **运输方式表**是行级数据——更新某行 vs 增减行需要不同 diff
- **节假日工作日历表**同上
- **About / Announcement 原文**的更新需要用户**完整粘贴新版**——不要让 Agent 逐字编辑英文原文（容易引入语法错误，且和 Hermes 的 `patch` 工具命名容易混）
- 如果用户说的事实跨多个字段，一个一个来，不要混改

### 第三步：更新 — diff 展示后落盘

用 diff 风格展示拟改动（保留旧值作为对比）：

```diff
## 2. 运营政策

### 处理时间
- 3 个工作日（从订单确认到发货）
+ 5 个工作日（从订单确认到发货）
- 例外说明：节假日延迟 2-3 天
+ 例外说明：节假日延迟 2-3 天；柴烧排窑期延迟 5-7 天
```

落盘按 SKILL.md §写入前的硬性约束执行（含确认、最后更新日期同步、修订日志追加）。

修订日志格式：

```
- 2026-XX-XX: <字段或块名> 更新（来源：<触发原因>）
```

例如：
```
- 2026-05-05: 处理时间从 3 天延长至 5 天，新增柴烧排窑期延迟说明（来源：用户主动告知）
- 2026-05-05: About 页面文本替换为新英文版本（来源：用户提供新版）
```

落盘后用一句话告诉用户改了哪部分、修订日志已更新。

## 与 BRAND.md 沉淀的关系

如果用户的反馈**同时**反映了品牌偏好 + 店铺事实变更（比如"我们品牌不该承诺这么短的处理时间，太赶了，改成 5 天"），分两步：

1. SHOP.md：直接把处理时间改成 5 天（事实更新）
2. BRAND.md：可能可以沉淀一条"运营节奏原则"（"小批量手作不追求工业化交付速度"）——按 distillation-brand.md 流程走

**不要混在一次 diff 里**——两个文件的修订日志要各自记录。

---

## 何时该问用户而不是直接改

下面这些场景**先问，不要直接改**：

- 用户的话有歧义（"运输再加一档"——是新增还是替换？）
- 改动会影响线上对外承诺（运输时长、退货政策、价格策略）—— 改前确认是否已经在 Etsy 后台同步
- 涉及法律或合规字段（税务、退款、IP）—— 用户措辞模糊时不要替他下结论
