# 订单履约 SOP

> 本 SOP 用于新订单 / 待发货订单的全链路执行检查。它不是客服话术；客服话术仍看 `order-handling.md`。承诺发货时间来源、买家语言等平台差异看目标平台 preset（`platforms/<platform>.md`，契约见 `platforms/platform-presets.md`）。

> **触达机制（读之前先懂这条）**：本 skill 请求触发，**不驻留后台定时器**。SOP 里所有"提醒 / 触达"都不是 skill 自己会定时发，而是——把待办状态写进 `Orders 订单` 表，让记录进入 `base-schema.md` 的待办视图（`临期/超期待发` / `待发货` / `待签收跟进` / `待复购触达`）。真正到点触达靠：① 运营每天看视图按待办处理；② 配了 Hermes cron 时由 cron 扫视图推给运营核查（`shared/preamble.md` §Hermes cron 例外）。下面写"设置提醒""到期触发"，指的就是"进了正确的待办视图等人 / cron 来取"，不是承诺自动发出。

> **签收事实来自 `logistics-tracking`**：凡涉及"是否已签收 / delivered"（阶段 5/6/7），用 `track` 查询为准（见 `../../logistics-tracking`），不要自己猜、也不要去承运商网页查（会撞号）。跟踪号录入后交 logistics-tracking 纳入跟踪。

## 适用触发

- 新订单录入 `Orders 订单` 表，需要给买家发送下单确认消息，确认所有商品细节。
- 订单状态进入 `待发货`。
- 用户问"这单下一步做什么 / 有没有漏什么 / 准备发货了吗"。
- 定制订单需要从确认内容走到制作、出货、跟进评价。
- 物流显示签收 / 已送达时，需要给买家发送收货关怀与评价引导消息。
- 签收后 30 天，需要给买家发送复购引导消息，并在有券码时附优惠券。

## 执行原则

1. 先查 `Orders 订单` 表、`Customers 客户` 表、`Products 商品` 表，再判断下一步；不要凭聊天记忆推进订单。
2. 对买家的承诺必须来自 `SHOP.md`，尤其是制作周期、发货方式、退换货和定制边界。
3. 平台承诺发货截止信息必须提取、写入订单记录，并落进 `临期/超期待发` / `待发货` 待办视图（= "发货前 1 天提醒"）；**承诺发货时间从哪取按目标平台 preset**（Etsy `Ship by ...` / `Ship tomorrow`；小红书 `promiseLastDeliveryTime`；其他平台见各自 preset）。不要只记录下单日期。
4. 需要买家确认的内容必须先出确认材料，再进入不可逆动作。
5. 证据文件进入素材库或可追溯链接；不要只留在聊天里。
6. 优惠券只在用户已提供券码 / 店铺政策允许时发送；没有券码时提醒用户补，不要编造。
7. 主动消息分四类：下单确认、发货前 1 天提醒、签收评价、30 天复购；每次都必须带订单具体细节，避免群发感。

## 阶段清单

### 0. 新订单录入与发货截止提醒

目标：订单产生时先把可见事实保存下来，并防止临近平台承诺发货时间漏发货。

检查：

- 从平台订单截图 / 后台 / API 导出中提取承诺发货时间，并换算为可执行的 `承诺发货日`。**来源字段按目标平台 preset**（Etsy `Ship by ...` / `Ship tomorrow`；小红书 `promiseLastDeliveryTime` / 后台承诺最晚发货时间；其他平台见各自 preset）。
- 将 `承诺发货日`、订单截图、订单号、客户、金额、下单日期、物流方式、`收件地区`（国家 / 州 / 省 / 市级）、商品/规格/定制文字写入订单记录。
- 落入待办视图（= "发货前提醒"，非后台定时器）：`承诺发货日` 一旦写入，记录自然进入 `待发货` 视图；`承诺发货日 - 1 天 <= 今天` 时进入 `临期/超期待发` 视图。运营每天看这两个视图，或由 Hermes cron 扫视图推送（见本文件头 §触达机制）。如果当前已到 / 已过 `承诺发货日 - 1 天`，**本轮就立即提醒用户**。
- 提醒内容必须包含：订单号（脱敏或完整按用户上下文）、买家姓名、商品、定制文字、平台承诺发货截止、当前 SOP 阶段、还缺哪些发货证据/字段。
- 如平台只给相对描述（如 tomorrow）且当前日期不明确，先用系统日期换算；仍不确定时在订单 `备注` 标记「需平台后台复核」。

完成条件：

- 订单记录已保存，并能从 `Orders 订单` 表查回 `承诺发货日`，记录已落入 `待发货`（必要时 `临期/超期待发`）视图。
- `SOP 阶段 = 内容确认`。

### 1. 内容确认

目标：确认买家要写什么、做什么、寄什么。

检查：

- 订单号、客户、SKU、数量已录入。
- 定制内容已记录到 `Orders 订单` 表的 `定制需求`。
- 新订单录入后，默认给买家发送下单确认消息，确认所有商品细节；不要只在定制订单里确认。
- 确认消息至少列明：商品名称 / SKU、数量、规格或尺寸、颜色 / 材质、定制文字或图案、礼物留言、收件场景、预计处理时间；不适用项可省略。
- 如涉及文字、姓名、日期、祝福语、颜色、尺寸、收件场景，逐项列给买家确认。
- 按 `order-handling.md` 的「下单确认」规则起草消息（买家语言按目标平台 preset）。

完成条件：

- `买家确认状态 = 已确认`，或用户明确标记不需要买家确认。
- 下单确认消息已草拟并给用户确认；用户发出后，将确认要点写入 `客服记录` 或 `备注`。

### 2. 制作 / 书写

目标：进入生产，但不提前承诺发货。

检查：

- 已确认制作内容。
- 承诺发货日未超期。
- 如制作中发现内容歧义，回到阶段 1，不继续制作。

完成条件：

- 成品完成。
- `Orders 订单` 表 更新 `SOP 阶段 = 待出货确认`。

### 3. 出货前拍照确认

目标：让用户确认最终寄出的东西完整、正确。

检查：

- 列出最终会寄出的所有物品：主商品、定制件、礼盒、卡片、说明书、赠品、保护材料。
- 如订单包含赠送信件 / 礼物信 / 手写卡片，必须单独确认信件内容：收件人称呼、正文、署名、语言、是否需要照抄买家原文、是否有敏感或不合适表达。
- 拍摄出货确认照片，至少包含所有物品的合照；定制文字需要可读；赠送信件如已写好，也应拍到可读内容或单独拍照留证。
- 照片链接写入 `确认照片链接`。

完成条件：

- 用户确认照片和清单无误。
- 如有赠送信件 / 礼物信 / 手写卡片，`出货清单` 或 `备注` 中已记录「信件内容已确认」及确认版本摘要。
- `出货清单` 已写入 `Orders 订单` 表。
- `SOP 阶段 = 待打包`。

### 4. 打包

目标：完成发货前包装，并保留必要的内部证据；打包视频用于可选的社交媒体素材或特殊订单留证，不是每单必须完成的客户通知动作。

检查：

- 先获得用户确认，再打包。
- 打包视频为**选择性录制**：可用于社交媒体内容、重要订单过程素材、或用户明确要求留证的订单；不是所有订单的必做项。
- 不默认要求把打包视频发给客户。只有当用户明确要给客户做打包过程通知，或订单存在高价值 / 易碎 / 纠纷风险等需要留证的情况，才起草客户消息；**若目标平台消息不支持视频**（见平台 preset，如 Etsy 站内信），改从视频里截取最多 3 张图片发送。
- 普通订单可把 `打包视频状态` 标记为 `不适用`（枚举：`必需 / 不适用 / 待补 / 已完成`），不得因此阻塞发货。
- 如选择录制视频，建议覆盖：商品入盒、保护材料、外箱封装、面单前的最终状态；视频链接可写入 `打包视频链接`，并更新 `打包视频状态 = 已完成`。
- 如视频用于社交媒体，应按素材库 / 社媒发布流程另行入库或排队，不作为订单履约完成的硬性条件。

完成条件：

- 包装完成；如未录制视频，`打包视频状态 = 不适用` 或留空，均不阻塞发货。
- 如本单明确需要视频留证，则 `打包视频状态 = 已完成` 且 `打包视频链接` 有值。
- `SOP 阶段 = 待发货`。

### 5. 发货与物流

目标：订单进入真实物流链路。

检查：

- 运输方式与 `SHOP.md` 对齐。
- 跟踪号已填，并交 `logistics-tracking` 纳入跟踪（`track add`）。
- 发货日期已填。
- 如果物流面单 / 打单系统 / 物流商账单已经给出实际支付金额，把 `实际运费成本` 按订单的 `核算币种`写入；没有证据时留空并在缺口里提示，不要用买家支付的运费或预估值代替。
- 如需要给买家发消息，按 `order-handling.md` 的订单咨询结构起草通知（买家语言按平台 preset）。

完成条件：

- `状态 = 已发货`。
- `跟踪号`、`发货日期` 已写入，跟踪号已纳入 `logistics-tracking`。
- 有实际物流成本证据时，`实际运费成本` 已写入；没有证据不阻塞发货，但必须标为待补字段。
- `SOP 阶段 = 待签收跟进`。
- 根据预计签收窗口填写 `签收跟进日期`；无法判断时不要猜，提示用户补物流时效。
- **签收以 `logistics-tracking`（`track query`）为准**：显示 delivered / 签收当天进入签收评价引导；把签收日回填到 `签收日期` 字段并将 `状态` 置为 `已签收`，再据此填写 `30天复购跟进日期 = 签收日期 + 30 天`。不要凭猜测当作已签收。

### 6. 签收即时评价引导

目标：确认买家收到商品，温和引导评价，但不要让消息像群发广告。

检查：

- 只有在 `logistics-tracking`（`track query`）显示签收 / delivered，或预计已送达且用户确认适合跟进后再发。
- 默认触发时间为签收当天或确认已送达当天；如 `SHOP.md`、COMMERCE_PLATFORM.md 平台规则或用户另有要求，以用户规则为准。
- 消息必须带一个订单具体细节，避免群发感。
- 重点是收货关怀 + 评价引导，不在这一步强推优惠券。

完成条件：

- 签收评价消息草稿整篇给用户确认。
- 用户发出后，摘要写入 `客服记录`。
- `签收评价消息状态` 更新为 `已发送`。

### 7. 签收后 30 天复购 / 优惠券

目标：在买家已使用一段时间后，邀请他们看看其他产品，并在有券码时发送优惠券。

检查：

- 仅在签收后满 30 天触发；如果签收日未知，先补签收证据或让用户确认日期。
- 先查 `Customers 客户` 表：客户标签、历史订单、偏好备忘，优先推荐与上次购买相关但不重复的产品方向。
- 消息必须提到上次购买的具体商品或使用场景，避免像群发广告。
- 优惠券只在用户已提供券码 / 店铺政策允许时发送；没有券码时，草稿里留占位并提醒用户补券码，不要编造。

完成条件：

- 30 天复购消息草稿整篇给用户确认。
- 用户发出后，摘要写入 `客服记录`。
- `优惠券发送状态` 更新为 `已发送` 或 `无券码`；如有单独字段，也更新 `30天复购消息状态`。

## 下一步判断

每次用户问某订单状态时，按下面顺序输出：

1. 当前 `SOP 阶段`。
2. 下一步动作。
3. 缺失证据或字段。
4. 是否需要用户确认。
5. 会写回 `Orders 订单` 表的字段预览。

如果字段缺失，不要继续编流程。先列缺口。
