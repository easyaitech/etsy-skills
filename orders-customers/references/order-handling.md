# 订单处理 SOP

> 写客服回复时按场景找对应 SOP。买家可见语言由 COMMERCE_PLATFORM.md 决定；所有回复**整篇展示给用户**确认后，由用户自己复制到目标平台后台 / 客服入口发送。

> ⚠️ 任何场景下，文案语调严格遵守 BRAND.md；政策段（处理时间、运输、退换货）严格引用 SHOP.md 原文。

---

## 场景 1：订单咨询（发货前）

**典型买家问**："Has this been shipped yet?" / "When will I receive it?"

**回复结构**：
1. 共情一句（"Thanks for checking in"）
2. 状态事实（"Your order is currently in production / has been shipped on X"）
3. 时间锚点（"You should receive it by X"——按 SHOP.md 运输时长算）
4. 邀请回头（"Let me know if anything comes up"）

**禁忌**：编造时间。如果不知道确切发货日，给区间，不要给点。

---

## 场景 2：定制咨询

**典型买家问**："Can you make this in [size/color/material]?"

**回复结构**：
1. 共情 + 简短说明你能否做（按 SHOP.md 定制政策）
2. 如能做：给定制起订量、加工时长、加价、能与不能的边界
3. 如不能做：解释原因（不绕弯，不假装能但暗示不愿——直接说工艺/产能边界）
4. 给替代方案（如有）

**禁忌**：为了拿单口头答应工艺上做不到的事。

---

## 场景 3：地址改写 / 急单

**典型买家说**："I just realized I gave the wrong address—can you change it?" / "Is there any way to ship faster, I need it by X"

**判断逻辑**：
- 未发货 + 简单改地址 → 改，回复确认
- 已发货 → **诚实告知**已发货，给跟踪号，建议联系本地物流改派
- 急单 → 按 SHOP.md 运输方式表，告知最快可选；不要承诺超出 SHOP.md 政策的时长

**禁忌**：私自承诺加急或免费升级运输。这种例外**先问主理人**（用户）。

---

## 场景 4：退换货

**典型买家说**："I'd like to return this" / "It arrived damaged"

**回复结构**（按 SHOP.md 退货政策走）：
1. 共情（不是机械道歉）
2. 询问情况（如果需要：要照片、订单号、问题描述）
3. 给出可执行下一步（按 SHOP.md：退款 / 重发 / 部分退）
4. 时间承诺（处理时长按 SHOP.md）

**关键边界**：
- 损坏但运输方责任 → 重发或退款，主动赔
- 买家不喜欢但物件本身正常 → 按 SHOP.md 退货政策处理（一般退货运费由买家承担）
- 定制款 → 按 SHOP.md 是否接受定制款退货执行

**禁忌**：条件反射式 "I'm so sorry, full refund"。这违反 BRAND.md 客服姿态——先共情、再了解、最后给方案。**不要在了解情况前承诺全退**。

---

## 场景 5：差评回复

**典型情况**：买家留 ≤ 3 星评价，可能附文字。

**回复策略（差评公开回复，所有人能看到）**：

1. **不为差评本身道歉**——道歉过快显得心虚
2. 表达**理解 + 关心**（"Hearing this is really useful for us. We care about how the piece feels in daily use."）
3. 简短**说明事实背景**（不辩解，不甩锅）—— 例如手作差异、烧窑色差是工艺特征
4. 给**实际下一步**（"Could we figure out a better outcome together? I'll reach out via the platform message thread"）
5. **不在公开评价里**透露解决方案细节（"will refund X" 这种 → 私信里说）

**回复的三个层次**（按 BRAND.md 客服姿态原则）：
- 共情 → 了解 → 方案

**严格禁忌**：
- 不要在公开回复里出现 "we offered them a refund but they refused" —— 显得记账
- 不要 ALL CAPS 或激动情绪
- 不要让差评成为 SEO 反例（不要在回复里无意中重复负面关键词）

**回复完成后**：
- 给客户加 `差评户` 标签
- 在 `Orders 订单` 表备注复盘原因（运输 / 工艺 / 期望管理）
- 如果反映了产品共性问题，提议跟用户讨论是不是该回流到 BRAND.md 边界条款（用 shop-foundation 沉淀）

---

## 场景 6：下单确认

**触发**：所有新订单录入后；尤其是包含定制文字、礼物留言、颜色 / 尺寸 / 材质选择、收件场景或多件商品的订单。

**回复结构**：
1. 感谢下单，语气简短温暖，不要像自动通知。
2. 列出需要确认的商品细节：商品名称 / SKU、数量、规格或尺寸、颜色 / 材质、定制文字或图案、礼物留言、预计处理时间。
3. 请买家检查并回复确认；如有任何细节需要调整，请在制作前告诉我们。
4. 如订单已明确无定制 / 无可变项，也可以写成「we’ll prepare it as ordered」并邀请他们指出错误。

**禁忌**：
- 不要只写 “Thanks for your order” 而不列细节。
- 不要暗示买家不确认就一定延误；表达为「为了确保一切准确」。
- 不要承诺 SHOP.md 没有写明的处理时间或加急。

---

## 场景 7：签收评价引导

**触发**：物流显示 delivered / signed，或用户确认买家应已收到商品。

**回复结构**：
1. 轻声确认收到：「I hope your [具体商品] arrived safely.」
2. 提到订单具体细节或使用场景，避免群发感。
3. 如他们喜欢，邀请他们在目标平台留评价；语气是邀请，不是索取。
4. 如有任何问题，请先回复消息，我们会一起处理。

**禁忌**：
- 不要在这一步塞优惠券或强销售。
- 不要写 “please leave us a 5-star review”；只能温和邀请 honest review / share your thoughts。
- 不要在未确认送达前催评价。

---

## 场景 8：签收后 30 天复购 / 优惠券

**触发**：订单签收后 30 天；适用于体验稳定后的复购触达。

**回复结构**：
1. 以之前购买的具体商品开头，表达希望它已经融入使用 / 送礼场景。
2. 基于客户标签、偏好或上次商品，推荐一个相关方向（例如配套款、礼盒套装、应季新品）。
3. 如用户提供了券码，附一句清晰但克制的优惠券说明；没有券码时不要编造，留占位提醒用户补。
4. 结尾保持轻，不制造紧迫感。

**禁忌**：
- 不要群发式 “Buy more now”。
- 不要每个客户都推同一件产品；至少按上次购买类别调整推荐方向。
- 不要发送未获授权或不存在的优惠券码。

---

## 场景 9：感谢信 / 主动问候

**触发**：
- 节日（按 SHOP.md 节假日工作日历对应的市场节日）
- 老客复购第 2、5、10 单等节点

**回复结构**：
1. 不要"营销"——按 BRAND.md 文案语调"像朋友分享"
2. 不要长，3-5 句够
3. 不要每次都附带促销码（廉价化品牌）—— 给"想看新东西吗"邀请就够了

**禁忌**：群发感觉。每封都该有一点该客户的具体信息（"the cup you got—I hope it's becoming part of your morning"）。

---

## 跨场景的通用规则

1. **先查 Base 拿上下文**：订单详情 + 客户标签 + 历史互动。不要在不知道背景的情况下回。
2. **政策事实 → 引 SHOP.md**：处理时间、运输、退换货——一字不差从 SHOP.md 抄；不要按记忆答。
3. **平台边界 → 引 COMMERCE_PLATFORM.md**：消息入口、自动化边界、买家语言和售后流程按目标平台配置；没配置就标注未知，不编造。
4. **语气 → 引 BRAND.md**：不温和、不克制 = 不像我们品牌。每条回复写完自己核一下"这像不像 BRAND.md 文案语调段描述的那种说话方式"。
5. **回复落地后回写 Base**：用户最终发出的版本要简要记到 `Orders 订单` 表的"客服记录"字段；如果这次互动反映客户特征，更新客户标签。
6. **纠正即沉淀**：用户改你的回复时，如果改的是语气/姿态层面的，提议沉淀到 BRAND.md（用 shop-foundation 流程）；如果改的是政策口径，提议更新 SHOP.md；如果改的是平台规则，提议更新 COMMERCE_PLATFORM.md。

## 小红书补充规则

目标平台是小红书时，先读 `xiaohongshu-orders.md`：

- 不使用 Etsy username、Etsy order number、Etsy message 这些字段名做判断。
- 订单详情导出前可能被消费者取消；遇到订单不存在 / 已取消，先记录取消状态，不继续推进履约。
- 售后通常进入小红书平台售后链路；草稿只写处理建议和中文客服话术，不越过平台流程承诺退款 / 补发。
- 收件人完整地址和电话默认不进 Base；只保留地区级信息、`openAddressId` 和打单证据。
