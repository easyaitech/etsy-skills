# 订单 Base + 客户 Base Schema

> 用 `lark-base` skill 在飞书云空间创建两个 Base：
> - `{店铺名}-订单库`
> - `{店铺名}-客户库`
>
> 两个 Base 的关联：订单 Base 的 `客户` 字段关联到客户 Base 的 `客户 ID` 主键。

---

## 客户 Base

### 核心字段（必建）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `客户 ID` | 单行文本（**主键**） | 内部唯一编号，建议格式 `C-{YYYY}-{NNNN}`，如 `C-2026-0001` |
| `Etsy Username` | 单行文本 | Etsy 上的 buyer username |
| `名称` | 单行文本 | 真实姓名（来自订单收货信息） |
| `首次下单日` | 日期 | 第一次下单的日期 |
| `订单数` | 数字 | 累计订单数（可用公式从订单 Base 自动算） |
| `累计金额 (USD)` | 数字（货币） | 累计成交额（公式自动算） |
| `客户标签` | 多选 | 见 `customer-tags.md`——只用统一标签体系，不要随手发明 |
| `偏好备忘` | 多行文本 | 该客户的特殊偏好（"喜欢素雅色"、"对运输延迟敏感"等） |

### 辅助字段（推荐）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `主要地址国家` | 单选 | 用于按地区分群 |
| `首次接触渠道` | 单选 | Etsy 搜索 / Instagram / 朋友推荐 / 老客复购 等 |
| `客服备忘` | 多行文本 | 历次客服互动的关键事件（差评经历、定制谈判等） |
| `最近互动日` | 日期 | 用于"沉默客户"召回筛选 |

### 字段不该塞的东西

- 邮箱 / 完整地址：敏感数据，留在 Etsy 后台即可，本 Base 不做完整副本
- 订单详情：进订单 Base，不要在客户 Base 重复

## 订单 Base

### 核心字段（必建）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `订单号` | 单行文本（**主键**） | Etsy 订单号 |
| `客户` | 关联（→ 客户 Base） | 关联到客户 ID |
| `下单日` | 日期 | Etsy 上订单创建日 |
| `状态` | 单选 | 待发货 / 已发货 / 已签收 / 退货中 / 已退款 / 有问题 |
| `总金额 (USD)` | 数字（货币） | 含运费 |
| `SKU 列表` | 多行文本 或 关联（→ 商品 Base） | 简单情况用文本 `SKU-001 ×2; SKU-007 ×1`；复杂场景另建订单明细子表 |
| `运输方式` | 单选 | 与 SHOP.md 运输方式表对齐 |
| `承诺发货日` | 日期 | 按 SHOP.md 处理时间倒推；超期是要警觉的指标 |
| `跟踪号` | 单行文本 | 发货后填 |
| `客服记录` | 多行文本 | 该订单相关的客服互动摘要（不存原文长篇） |

### 辅助字段（推荐）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `定制需求` | 多行文本 | 如果是定制订单，记录买家提出的定制点 |
| `备注` | 多行文本 | 任何特殊情况：地址写错改地址、急单、礼物寄送等 |
| `差评 / 投诉` | 单选 | 无 / 有差评 / 有投诉 / 有差评+投诉 |
| `评价记录` | 多行文本 | 买家评价文本摘要 + 星级（如有） |

### SOP 状态字段（推荐，订单量起来后建）

这些字段用于 `order-fulfillment-sop.md`，防止履约流程只留在聊天里。新店可以先不建全；一旦开始处理定制 / 高价值 / 易碎订单，建议补齐。

| 字段名 | 类型 | 说明 |
|---|---|---|
| `SOP 阶段` | 单选 | 内容确认 / 制作中 / 待出货确认 / 待打包 / 待发货 / 已发货 / 待签收跟进 / 已完成 |
| `买家确认状态` | 单选 | 不需要 / 待确认 / 已确认 / 需修改 |
| `下单确认消息状态` | 单选 | 待草拟 / 已草拟 / 已发送 / 不需要 |
| `出货清单` | 多行文本 | 最终寄出的物品清单：主商品、定制件、礼盒、卡片、赠品、保护材料等 |
| `确认照片链接` | 链接 / 附件 | 发给用户或买家确认的出货前照片 |
| `打包视频状态` | 单选 | 必需 / 不适用 / 待补 / 已完成 |
| `打包视频链接` | 链接 / 附件 | 打包过程视频；`打包视频状态 = 已完成` 时必填 |
| `发货日期` | 日期 | 实际交给物流的日期 |
| `签收跟进日期` | 日期 | 预计可提醒收货 / 评价的日期；无法判断时留空并提示用户补物流时效 |
| `评价跟进状态` | 单选 | 未到时间 / 待跟进 / 已草拟 / 已发送 / 不跟进 |
| `签收评价消息状态` | 单选 | 未到时间 / 待草拟 / 已草拟 / 已发送 / 不跟进 |
| `30天复购跟进日期` | 日期 | 签收后 30 天的复购触达日期；签收日未知时留空 |
| `30天复购消息状态` | 单选 | 未到时间 / 待草拟 / 已草拟 / 已发送 / 不跟进 |
| `优惠券码` | 单行文本 | 30 天复购消息要发送的券码；没有时不要编造 |
| `优惠券发送状态` | 单选 | 无券码 / 待发送 / 已发送 / 不发送 |

### 视图建议

- **待发货** — 状态=待发货，按下单日升序
- **超期未发** — 状态=待发货 且 承诺发货日 < 今天 — **每天都该看这个视图**
- **履约卡住** — `SOP 阶段` 不为空 且 阶段不在 {已发货, 待签收跟进, 已完成}，按承诺发货日升序
- **待签收跟进** — `签收评价消息状态` ∈ {待草拟, 已草拟}、`评价跟进状态` = 待跟进，或 `签收跟进日期` <= 今天
- **待复购触达** — `30天复购消息状态` ∈ {待草拟, 已草拟} 或 `30天复购跟进日期` <= 今天
- **本月** — 下单日在当月，按金额倒序
- **有问题** — 状态∈{退货中, 有问题} 或 差评/投诉 ≠ 无
- **回头客订单** — 客户的"订单数" > 1

## 跨表关联与公式

- 客户 Base 的 `订单数` 公式：`COUNTA(关联订单)` 或类似
- 客户 Base 的 `累计金额` 公式：`SUM(关联订单.总金额)`
- 订单 Base 的 `客户` 是关联字段——录新订单时先在客户 Base 找 / 建对应客户行，避免重复客户
