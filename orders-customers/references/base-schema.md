# `Orders 订单` 表 + `Customers 客户` 表 Schema

> 用 `lark-base` skill 在店铺总 Base 内创建或补齐两张表：
> - `Orders 订单`
> - `Customers 客户`
>
> 两张表的关联：`Orders 订单` 表的 `客户` 字段关联到 `Customers 客户` 表的 `客户 ID` 主键。
> 字段默认平台中性。历史 Etsy 工作区可以保留旧字段名，但新建表优先使用下面的通用字段；平台专属字段放在对应分组。

---

## `Customers 客户` 表

### 核心字段（必建）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `客户 ID` | 单行文本（**主键**） | 内部唯一编号，建议格式 `C-{YYYY}-{NNNN}`，如 `C-2026-0001` |
| `销售平台` | 单选 / 多选 | Etsy / 小红书店铺 / Shopify / 淘宝 / 天猫 / 抖店等；取值以 COMMERCE_PLATFORM.md 为准 |
| `平台客户 ID` | 单行文本 | 平台侧可见的 buyer id / user id / open id；没有稳定 ID 时留空 |
| `平台用户名 / 昵称` | 单行文本 | Etsy username、小红书昵称或其他平台展示名；没有时留空 |
| `名称` | 单行文本 | 真实姓名（来自订单收货信息）；没有时留空 |
| `首次下单日` | 日期 | 第一次下单的日期 |
| `订单数` | 数字 | 累计订单数（可用公式从 `Orders 订单` 表自动算） |
| `累计金额` | 数字（货币） | 累计成交额（公式自动算）；币种按 COMMERCE_PLATFORM.md / SHOP.md |
| `客户标签` | 多选 | 见 `customer-tags.md`——只用统一标签体系，不要随手发明 |
| `偏好备忘` | 多行文本 | 该客户的特殊偏好（"喜欢素雅色"、"对运输延迟敏感"等） |

### 辅助字段（推荐）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `主要地址国家 / 地区` | 单选 | 用于按地区分群；不保存完整地址 |
| `首次接触渠道` | 单选 | 取值见 `customer-tags.md` §`首次接触渠道` 字段取值（Etsy 搜索 / 小红书 / Instagram / 朋友推荐 / 他渠道；随启用平台扩展） |
| `客服备忘` | 多行文本 | 历次客服互动的关键事件（差评经历、定制谈判、售后沟通等） |
| `最近互动日` | 日期 | 用于"沉默客户"召回筛选 |

### 字段不该塞的东西

- 邮箱 / 手机号 / 完整地址：敏感数据，留在平台后台 / ERP / 打单系统即可，本 Base 不做完整副本
- 订单详情：进 `Orders 订单` 表，不要在 `Customers 客户` 表重复

## `Orders 订单` 表

### 核心字段（必建）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `订单号` | 单行文本（**主键**） | 内部统一订单号；可直接使用平台订单号，也可用 `{平台}-{平台订单号}` 防撞 |
| `销售平台` | 单选 | Etsy / 小红书店铺 / Shopify / 淘宝 / 天猫 / 抖店等；取值以 COMMERCE_PLATFORM.md 为准 |
| `平台订单号` | 单行文本 | 平台侧订单号 / order id |
| `平台包裹 ID` | 单行文本 | 平台以包裹为履约单元时填写；如小红书 `packageId` |
| `客户` | 关联（→ `Customers 客户` 表） | 关联到客户 ID |
| `下单日` | 日期 | 平台订单创建日 |
| `状态` | 单选 | 待发货 / 已发货 / 已签收 / 退货中 / 已退款 / 有问题 |
| `平台订单状态` | 单行文本 / 单选 | 平台原始状态；不要覆盖通用 `状态`，两者并存便于对账 |
| `总金额` | 数字（货币） | 含运费；币种按 COMMERCE_PLATFORM.md / SHOP.md |
| `SKU 列表` | 多行文本 或 关联（→ `Products 商品` 表） | 简单情况用文本 `SKU-001 ×2; SKU-007 ×1`；复杂场景另建订单明细子表 |
| `运输方式` | 单选 | 与 SHOP.md / COMMERCE_PLATFORM.md 运输方式表对齐 |
| `承诺发货日` | 日期 | 按平台承诺发货时间或 SHOP.md 处理时间倒推；超期是要警觉的指标 |
| `跟踪号` | 单行文本 | 发货后填 |
| `快递公司` | 单行文本 | 发货后填；平台有编码时编码进平台字段 |
| `客服记录` | 多行文本 | 该订单相关的客服互动摘要（不存原文长篇） |

### 辅助字段（推荐）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `收件地区` | 单行文本 / 多行文本 | **平台中性**的目的地，只存国家 / 州 / 省 / 市级（用于按地区分群、报关粗判、SOP 阶段 0 记录）；**不存完整地址 / 门牌 / 电话**——完整地址留平台后台 / 打单系统。小红书有更细的 `小红书收件地区` 字段，两者择一即可 |
| `定制需求` | 多行文本 | 如果是定制订单，记录买家提出的定制点 |
| `备注` | 多行文本 | 任何特殊情况：地址写错改地址、急单、礼物寄送等 |
| `差评 / 投诉` | 单选 | 无 / 有差评 / 有投诉 / 有差评+投诉 |
| `评价记录` | 多行文本 | 买家评价文本摘要 + 星级（如有） |
| `售后状态` | 单选 | 无 / 申请中 / 处理中 / 已解决 / 已拒绝 / 待确认 |
| `平台字段 JSON` | 多行文本 | 其他平台专属订单字段；仅保存有证据或已确认的字段 |

## 平台专属字段组（按启用平台建对应组）

> 上面的通用字段对任何平台都建。**平台特有**的结构化字段不进通用区，而是按平台单独成组——组名用平台前缀（`小红书…` / `亚马逊…`），由该平台 preset（`platforms/<platform>.md`，契约见 `platforms/platform-presets.md`）定义。原则：
> - 启用某平台、且该平台 preset 定义了字段组 → 建表时一并加建该组（字段可先为空，但要预留，避免把平台状态塞进 `备注`）。
> - 平台暂时没必要细化结构化字段 → 先用通用 `平台字段 JSON`（订单）兜底，仅保存有证据或已确认的字段。
> - 加新平台（如亚马逊）想细化时，照下面小红书的范式新增一个 `亚马逊…` 字段组，并在该平台 preset 登记。

### SOP 状态字段（推荐，订单量起来后建）

这些字段用于 `order-fulfillment-sop.md`，防止履约流程只留在聊天里。新店可以先不建全；一旦开始处理定制 / 高价值 / 易碎订单，建议补齐。

| 字段名 | 类型 | 说明 |
|---|---|---|
| `SOP 阶段` | 单选 | 内容确认 / 制作中 / 待出货确认 / 待打包 / 待发货 / 已发货 / 待签收跟进 / 已完成 |
| `买家确认状态` | 单选 | 不需要 / 待确认 / 已确认 / 需修改 |
| `下单确认消息状态` | 单选 | 待草拟 / 已草拟 / 已发送 / 不需要 |
| `出货清单` | 多行文本 | 最终寄出的物品清单：主商品、定制件、礼盒、卡片、赠品、保护材料等 |
| `确认照片链接` | 链接 / 附件 | 发给用户或买家确认的出货前照片 |
| `打包视频状态` | 单选 | 必需 / 不适用 / 待补 / 已完成 |
| `打包视频链接` | 链接 / 附件 | 打包过程视频；`打包视频状态 = 已完成` 时必填 |
| `发货日期` | 日期 | 实际交给物流的日期 |
| `签收日期` | 日期 | 实际签收 / delivered 日期；以 `logistics-tracking`（`track query`）的签收事实为准回填，不要手猜。签收时同步把 `状态` 置为 `已签收`；`30天复购跟进日期` 从本字段 +30 天推算 |
| `签收跟进日期` | 日期 | 预计可提醒收货 / 评价的日期；无法判断时留空并提示用户补物流时效 |
| `签收评价消息状态` | 单选 | 未到时间 / 待草拟 / 已草拟 / 已发送 / 不跟进——**签收评价跟进的唯一状态字段**（不要再单建「评价跟进状态」，避免两字段含义重叠对不齐） |
| `30天复购跟进日期` | 日期 | 复购触达日期 = `签收日期` + 30 天；`签收日期` 未知时留空 |
| `30天复购消息状态` | 单选 | 未到时间 / 待草拟 / 已草拟 / 已发送 / 不跟进——复购消息**整体**的状态 |
| `优惠券码` | 单行文本 | 30 天复购消息要发送的券码；没有时不要编造 |
| `优惠券发送状态` | 单选 | 无券码 / 待发送 / 已发送 / 不发送——**仅**跟踪券码这一项（从属于 `30天复购消息状态`，回答"券发了没"，不重复消息整体状态）|

### 视图建议

- **待发货** — 状态=待发货，按下单日升序
- **临期/超期待发** — 状态=待发货 且 `承诺发货日 - 1 ≤ 今天`（含明天到期的临期单 + 已超期单）— **每天都该看这个视图**；这就是"发货前 1 天提醒"的落点
- **履约卡住** — `SOP 阶段` 不为空 且 阶段不在 {已发货, 待签收跟进, 已完成}，按承诺发货日升序
- **待签收跟进** — `签收评价消息状态` ∈ {待草拟, 已草拟}，或 `签收跟进日期` <= 今天
- **待复购触达** — `30天复购消息状态` ∈ {待草拟, 已草拟} 或 `30天复购跟进日期` <= 今天
- **本月** — 下单日在当月，按金额倒序
- **有问题** — 状态∈{退货中, 有问题} 或 差评/投诉 ≠ 无
- **回头客订单** — 客户的"订单数" > 1

**平台专属视图**（启用对应平台时按 preset 契约 #12 追加，通用视图不动）：

- **小红书异常** — 销售平台=小红书店铺 且 `小红书异常备注` 非空，或取消 / 售后状态非空
- 其他平台（如亚马逊）启用时，按各自 preset 追加 `{平台}异常` 类视图

## 跨表关联与公式

- `Customers 客户` 表的 `订单数` 公式：`COUNTA(关联订单)` 或类似
- `Customers 客户` 表的 `累计金额` 公式：`SUM(关联订单.总金额)`
- `Orders 订单` 表的 `客户` 是关联字段——录新订单时先在 `Customers 客户` 表找 / 建对应客户行，避免重复客户
