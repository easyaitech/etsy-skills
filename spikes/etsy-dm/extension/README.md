# Etsy DM 助手 —— 浏览器插件 v0.1(产品命脉验证)

产品真正的命脉:**content-script 扩展能不能在租客的 HubStudio 里跑、读 Etsy 私信、把草稿填进回复框。** 跑不通,这功能就没意义(你的原话)。所以先用最小 v0.1 把这条路验死。

v0.1 **只验机制**:背景里是 stub 草稿(不接真脑)、**绝不自动发送**。

## 装进 HubStudio
1. HubStudio → 你的 Etsy 环境 → 扩展管理 → **加载本地扩展**,选这个目录(`spikes/etsy-dm/extension/`)。
   (具体入口你比我清楚;HubStudio 支持给环境加扩展。MV3 + 内核 Chrome 100 应该能跑,跑不起来就是第 1 关失败。)
2. 在那个环境里打开 Etsy → Messages(私信页)。

## 看四件事(make-or-break 清单)
- [ ] 右下角弹出"Etsy DM 助手 v0.1 已加载 ✅" → **扩展能在 HubStudio 里跑**(最硬的一关,过了产品路就活)。
- [ ] 面板显示"读到最新私信(N 字)✅" → **能读到私信**(读不到 → 调 `content.js` 选择器,但不影响第 1 关结论)。
- [ ] 点"生成草稿并填入" → 回复框出现 stub 草稿 → **能写回回复框**。
- [ ] 全程**没有自动发送** → 人审闸门在。

四个都 ✅ = 产品这条路活了,接真脑 + 真起草即可。
第 1 关 ❌(HubStudio 不让跑扩展)= 整条产品路得换思路,这正是要最先知道的。

## 验过机制后:接真脑
改 `background.js` 的 `BRAIN_URL`:
- 自用:你本地 Hermes 起草接口,`http://127.0.0.1:<port>/draft`,契约 `POST {message} → {draft}`。
- 产品:云端同契约服务。

content.js 里读私信 / 找回复框的选择器是粗的,按 Etsy 真实 DOM 调 —— 那是第 1 关过了之后的活。
